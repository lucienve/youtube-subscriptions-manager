## 2024-04-09 - CSV/Formula Injection Bypass via Whitespace
**Vulnerability:** The existing `escapeFormula` function intended to prevent CSV/Formula injection by prefixing a single quote (`'`) to strings starting with `=`, `+`, `-`, or `@`. However, it did not account for leading whitespace or zero-width characters. An attacker could bypass the filter by supplying `  =IMPORTXML(...)` or `\u200B=CMD(...)`.
**Learning:** Security regexes for formula injection prevention must consume/ignore any leading whitespace and special invisible spacing characters, because Google Sheets (and other spreadsheet tools) will trim them and execute the formula anyway.
**Prevention:** Use a regex like `/^[\s\u200B\uFEFF\xA0]*[=+\-@]/` when checking strings to be written into Google Sheets, ensuring no leading characters can hide the dangerous prefix.

## 2024-05-24 - Prototype Pollution in Google Apps Script via Sheet Data
**Vulnerability:** In `Workflow_Predict.js`, dynamic data read from a spreadsheet (like channel names or keywords) was directly used as keys for standard JavaScript objects (`{}`). An attacker could place the payload `__proto__` into the spreadsheet. Since the environment is the V8 runtime, this modifies the global `Object.prototype`, causing widespread prototype pollution and potentially disrupting the execution of all server-side apps script functions in the project.
**Learning:** Never trust data from a spreadsheet when building map-like data structures where keys are derived from the data. The V8 runtime in Google Apps Script is fully susceptible to standard JavaScript prototype pollution attacks.
**Prevention:** Always initialize maps or dictionaries intended to hold arbitrary keys using `Object.create(null)` instead of standard object literals (`{}`). Additionally, explicitly cast data read from sheets to strings via `String()` before using them as keys, as spreadsheet data could sometimes be parsed into unexpected objects.
